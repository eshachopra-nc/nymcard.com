# CLAUDE.md

This document is read by Claude Code (the CLI agent that writes code in this repository) before any code is generated, before any file is edited, and before any pull request is opened. It is the gatekeeper between design intent and implementation.

Treat this document the same way you treat compile errors: if you would do something that contradicts it, stop and ask.

---

## Read order on every session

When you (Claude Code) start a session in this repository, read these documents in order, before doing anything else:

1. **This file (`CLAUDE.md`)** — rules of engagement.
2. **`../01-design-system/design-system.md`** — what the visual outcome must be. Authority over visual decisions.
3. **`../01-design-system/tokens.json`** — machine-readable token values. Authority over specific values. Mirrored at build time into `./lib/tokens.ts` (typed TypeScript constants for non-Tailwind contexts: Framer Motion, inline styles, SVG attributes) and into the `@theme` block in `app/globals.css` (Tailwind utility classes); both are generated by `scripts/build-tokens.js` — never hand-edit them.
4. **`../00-strategy/stack-decision.md`** — which libraries do which jobs. Authority over technical choices.
5. **The relevant Notion copy page** — the locked copy for the page you're working on. Authority over text content.

You do not need to re-read these mid-session unless they have been updated. You do need to read them at the start of every session.

## What this repository is and is not

This repository is the **build** for nymcard.com. Specifically: it converts approved designs into Next.js code, faithfully and without visual invention. Design happens upstream (in Claude Design and Stitch, captured as static artefacts in `../05-handoff/`). Copy happens upstream (in Notion). Your job is implementation fidelity, not creative direction.

You are not:
- Designing the visuals. The visuals come from `../05-handoff/`.
- Writing the copy. The copy comes from Notion (page IDs in the locked copy section below).
- Picking libraries. The stack is in `stack-decision.md`.
- Inventing tokens. The tokens are in `tokens.json`.

You are:
- Translating approved designs into clean, accessible, performant React + Tailwind code.
- Wiring up motion using Framer Motion against the motion tokens.
- Connecting copy from Notion via MCP, never hardcoded inline.
- Asking clarifying questions when handoff artefacts are ambiguous.

## Hard rules

These are the rules where drift caused the previous build to fail. They are non-negotiable. If a request would require breaking one, stop and ask first.

### Rule 1 — Never invent visual decisions.

If the handoff artefact in `../05-handoff/` does not specify a value (colour, spacing, radius, motion timing, layout), the answer is one of:

1. The token in `tokens.json`.
2. A clarifying question to the user.

It is never your own guess. "Looks about right" is not acceptable.

### Rule 2 — Never hardcode copy.

All marketing copy comes from Notion. The locked Notion page IDs are listed in the "Locked copy sources" section below. Use the Notion MCP to fetch copy at build time or via a fetch-and-cache pattern. Placeholder copy is acceptable for component-level work that is not section-specific; if you use placeholder copy, mark it clearly with a `// TODO: fetch from Notion` comment.

When you fetch copy, fetch it exactly as written. Do not edit, tighten, or "improve" the copy. The copy is approved at a different layer.

### Rule 3 — Use only the libraries in `stack-decision.md`.

The stack is locked. Adding a new library is a deliberate decision, not an in-session move. If you reach a point where you think a new library is needed, stop and propose it to the user with reasoning. Do not install it unilaterally.

The libraries explicitly banned in `stack-decision.md` ("What's explicitly out") are not options under any circumstance. If you find yourself reaching for 21st.dev, GSAP, Lottie, three.js, CSS-in-JS, a state library, or a CMS — stop. They are out for documented reasons.

### Rule 4 — Read tokens via `tokens.ts`, not inline values.

The `scripts/build-tokens.js` script generates `lib/tokens.ts` from `tokens.json`. In component code, import from `tokens.ts`:

```tsx
import { tokens } from '@/lib/tokens';

<motion.div
  transition={{
    duration: parseInt(tokens.motion.duration.slow) / 1000,
    ease: tokens.motion.easing.out,
  }}
/>
```

For values that map to Tailwind classes (colours, spacing, radii, typography), use the Tailwind utility classes generated from the same tokens:

```tsx
<button className="bg-brand-navy text-on-brand rounded-button px-7 py-2.5">
  Talk to sales
</button>
```

Never use a raw hex code, raw pixel value, or raw motion duration in component code. If a token doesn't exist for what you need, stop and ask whether the token should be added — don't bypass the system.

### Rule 5 — Respect the third-person company voice rule.

The website addresses the buyer as "you" and refers to the company as "NymCard" by name. Never "we," "our," "us" referring to NymCard. This applies to button text, navigation, error messages, empty states, micro-copy — anywhere copy might be written outside Notion. (Notion copy is already correct; this rule covers UI strings you might author.)

This rule comes from the NymCard MCD §9.6 with a website-specific override allowing "you/your." Detailed in the copywriting skill (`../00-strategy/` if present, otherwise reference the MCD via Notion).

### Rule 6 — Motion respects `prefers-reduced-motion`.

Every motion in the codebase either:
- Wraps in a `useReducedMotion()` check from Framer Motion, or
- Uses Framer Motion's built-in respect for `prefers-reduced-motion` via the `MotionConfig` provider in the root layout.

Disabled motion categories: ambient, kinetic, cinematic. Preserved: responsive motion under 200ms (hover, focus). See `tokens.json` → `motion.accessibility`.

### Rule 7 — Server components by default. Client components only when needed.

Next.js App Router defaults to server components. Add `'use client'` only when a component genuinely needs client-side capability (interactivity, browser APIs, hooks like `useState` or `useEffect`, or Framer Motion). Pushing everything to client components negates the App Router's performance benefit.

A useful heuristic: if a component is static content + a Framer Motion wrapper, the wrapper is the client component and the content can be passed as children from a server component.

### Rule 8 — Accessibility is not optional.

Every interactive element must:
- Have a keyboard focus state (handled by `tokens.shadow.ring.default`).
- Be reachable via tab navigation.
- Have an accessible label (visible text or `aria-label`).
- Have a hit area of at least 44×44px on touch devices (CTAs already meet this with 10px 28px padding + button font size).

Use shadcn/ui primitives where they exist — they handle most accessibility correctly out of the box. When building bespoke marketing components, mirror the same patterns.

## Soft rules (judgement calls)

These are rules that almost always apply but have legitimate exceptions. Use judgement; when in doubt, ask.

- **Keep components in `components/sections/` focused.** A section component should render one section of one page. Don't compose multiple sections into a single component. The Hero is one component; the bento ProductsGrid is another.
- **Co-locate styles, not types.** Tailwind classes live inline in JSX. TypeScript types live in shared files (`lib/types.ts` or co-located `*.types.ts` when truly local).
- **Loading states for data-fetched content.** Notion copy fetches need a sensible loading state — skeleton or layout shift prevention. Use shadcn's `Skeleton` primitive.
- **No `any` types in TypeScript.** If you need to escape the type system, use `unknown` and narrow. `any` is a code smell that almost always points to a missed type definition.
- **Tests are not required for the marketing site.** The site is mostly static content with motion. Component-level unit tests add little value here. If you find yourself wanting to write one, ask first — there may be a structural issue the test is masking.

## How to handle ambiguity

The previous build failed in part because Claude Code, when faced with ambiguity, invented an answer rather than asking. That is the failure mode this section exists to prevent.

When you encounter ambiguity in a handoff artefact, copy, or token:

1. **First, check if the answer is already in the source-of-truth documents** (`design-system.md`, `tokens.json`, `stack-decision.md`). Most of the time it is.
2. **If not, frame a specific question and ask the user.** Do not make a "best guess." A specific question that takes 30 seconds to answer is always better than a fix later.
3. **If the user is unavailable and you must proceed, mark the assumption clearly** in a code comment with `// ASSUMPTION:` prefix, and surface it in the PR description. The user can correct it during review.

Examples of good clarifying questions:
- "The handoff shows two buttons in the hero, but doesn't specify the secondary CTA copy. The current homepage Notion has 'Explore nCore' as secondary. Use that?"
- "The bento grid handoff shows the Card Issuing card in a 2x1 span. The token system supports this, but it changes the grid asymmetry from what's in the visual direction. Confirm this is intentional?"
- "The motion spec for the nCore module assembly is 'cinematic' but doesn't specify which elements assemble in what order. Sketch the order or shall I propose one?"

Examples of bad responses to ambiguity:
- Picking a hex code that's close to the design and shipping it.
- Picking an arbitrary section padding because "120px looks right."
- Writing your own copy because the Notion fetch failed.
- Installing a new library because "the design needs animation and Framer Motion doesn't do exactly this."

```
│   ├── tokens.ts                   ← Auto-generated from tokens.json (do not edit by hand)
│   ├── motion-tokens.ts            ← Helper for Framer Motion to read motion tokens
│   ├── notion.ts                   ← MCP client for fetching copy

├── scripts/
│   └── build-tokens.js             ← Reads tokens.json, generates tokens.ts and tailwind config
├── tokens.json                     ← Source of truth (synced from ../01-design-system/)
├── tailwind.config.js              ← Generated (do not edit by hand)
├── next.config.js
├── tsconfig.json
├── package.json
└── CLAUDE.md                       ← This file
```

Rules about the folder structure:

- **Never edit `lib/tokens.ts` by hand.** It's generated. Run `npm run build:tokens` to regenerate.
- **Never edit `tailwind.config.js` by hand** for token-related values. The token-related sections are generated. If you need to add Tailwind plugins or non-token config, edit only the non-generated sections (clearly marked with comments).
- **Bespoke SVG visuals live in `components/visuals/`**, not `public/`. They're React components, not static assets.
- **Static assets (raster images, logos) live in `public/`** and are referenced via Next.js `<Image>`.

## Common tasks — how to handle them

### Building a new section

1. Read the handoff artefact for the section (`../05-handoff/`).
2. Identify which `tokens.json` values apply: spacing, colours, radii, motion timings.
3. Identify which shadcn primitives apply (Button, etc.) and which bespoke components are needed.
4. Fetch the section copy from Notion via `lib/notion.ts`.
5. Build the section as a component in `components/sections/[SectionName].tsx`.
6. Wire motion using Framer Motion, reading durations and easings from `lib/tokens.ts`.
7. Test responsive behaviour at the breakpoints in `tokens.json` → `grid.breakpoints`.
8. Verify `prefers-reduced-motion` is respected.

### Adding a new shadcn component

```bash
npx shadcn-ui@latest add [component-name]
```

The component lands in `components/ui/`. Modify it to match `tokens.json` if shadcn's defaults don't align (most often: radius, colour palette).

### Adding a new icon

- System / UI icon (chevron, close, search, arrow): use Lucide.

```tsx
import { ArrowRight } from 'lucide-react';
```

- Brand / product icon (Card Issuing, Embedded Lending, etc.): bespoke SVG in `components/visuals/icons/` or `public/icons/` depending on whether it needs to be a React component or static asset.

### Adding a new page

1. Create `app/[route]/page.tsx`.
2. Use the existing `RootLayout` for shared chrome (nav, footer).
3. Compose the page from section components.
4. Fetch copy from the relevant Notion page (add the page ID to "Locked copy sources" in this file).
5. Update sitemap (`app/sitemap.ts`) if it exists.

### Updating a token

1. Edit `../01-design-system/tokens.json`.
2. Copy the updated file into the build folder root (or set up a sync script — preferred).
3. Run `npm run build:tokens` to regenerate `lib/tokens.ts` and the Tailwind config.
4. Test that affected components reflect the change.

Token updates are deliberate decisions. Do not update tokens to fit a one-off design need — the token system should not bend to a single component.

## What to do if something feels wrong

Three escalation levels:

**Level 1 — Internal check.** Re-read the relevant source-of-truth document. Most of the time, the answer is already there and you missed it.

**Level 2 — Frame a question.** Write the question as specifically as possible. Quote the conflict ("the handoff says X but `design-system.md` §8.5 says Y"). Ask the user.

**Level 3 — Stop and flag.** If you cannot proceed without breaking a rule in this document, stop. Do not make progress at the cost of fidelity. The user would rather wait for a clarifying answer than rebuild from a wrong assumption.

## What "good" looks like in this codebase

A successful contribution to this codebase produces code that:

- Looks visually identical to the handoff artefact at the breakpoints in `tokens.json`.
- Reads tokens via `lib/tokens.ts` and Tailwind utility classes only — no raw values in component code.
- Fetches copy from Notion — no hardcoded marketing strings.
- Uses motion that maps to a section of `design-system.md` §9 — no improvised animations.
- Is accessible by default — keyboard navigable, screen-reader friendly, `prefers-reduced-motion` respecting.
- Adds zero new dependencies unless the user explicitly approved them.
- Is reviewable in a single PR — focused scope, clear commit messages, no incidental changes.

A failed contribution is one that ships but introduces drift — a hardcoded hex, a copy string that wasn't in Notion, a library that wasn't in `stack-decision.md`. These don't fail visually at PR time; they fail months later when the team can't keep the site consistent.
